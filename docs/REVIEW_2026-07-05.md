## 2. Weaknesses and technical debt (stated honestly)

1. **Performance is "good" but not "the best"**: compute-bound (5% difference between f32 and f64).
   Still AoS scalar. Room for 2-3x with SoA + SIMD, 10-100x with GPU.
2. **2D only**: the commercial primary use case (3D) is not yet started. D3Q19 requires
   generalizing the core design.
3. **Multiphase constraints**: single-component only (two-component MCMP not yet implemented →
   RT instability T12 not reached). Density ratio ~16 (can be raised to 100+ with CS-EOS).
   Narrow range on the wetting side of contact angle.
4. **Open boundary quality**: zero-gradient outflow has pressure reflection (ratio 11). Zou-He
   outlet has a staggered layer (4-cell decay). Both can be improved with a characteristic BC.
5. **Cylinder uses a staircase approximation**: Schäfer-Turek's band is wide (±6%). Can be
   tightened to ±1-2% with a curved boundary (interpolated bounce-back).
6. **The GUI eraser rebuilds the flow** (flow stops). Live editing would be possible if core
   had clear_solid. Unit conversion (physical units ⇔ lattice units) is also not yet provided,
   leaving beginners with "what is ν=0.02?" unanswered.
7. **MCP is synchronous execution only** (a long run blocks). Progress notifications and
   asynchronous jobs are not yet supported.
8. **No CI** (local only). Once a remote repository is decided, attach GitHub Actions.

- **8a. MCMP (two-component Shan-Chen)**: couples two distribution functions over shared
  solid/geometry. Validation: phase separation, mass conservation of droplet coalescence,
  **RT instability linear growth rate ±20% (T12)**

  codex validation via cylinder wake / channel

### Phase 9: Performance leap (to a level that can claim to have "solved" the trade-off)
- **9a. SoA layout + explicit SIMD** (std::simd or wide): 2-3x at f32
- **9b. Fused collide-stream** (40% reduction in memory traffic)
- **9c. (evaluation) wgpu compute backend**: a GPU implementation behind the same Simulation
  API. 10-50x in 2D, prerequisite technology for 3D
- Update the benchmark table in PERFORMANCE.md, raise the GUI's resolution ceiling

  monolithic execution, as T13)
- Validation: 3D TGV, 3D Poiseuille, sphere drag
- GUI: cross-section display + isosurface (or accept the decision that 3D is CLI/agent-only)

## 4. Judgment on execution order

From a user-value standpoint, **Phase 8 (fulfilling the physics promise) → 9 (performance) → 10 (3D)**
is recommended. T12 (RT) is an explicit requirement of the original goal, so it takes top priority.
9a/9b can proceed in parallel with 8's codex validation.
